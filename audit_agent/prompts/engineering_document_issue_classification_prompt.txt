你是一名工程资料审计与内控专家，熟悉工程管理、合同管理、成本管理及资料归档规范。

下面给出一条【工程资料检查会议纪要中的问题描述】，请你依据以下【工程资料问题分类体系】，对该问题进行【唯一且排他的分类判断】。

——————————————————
【工程资料问题分类体系（审计级定义）】
——————————————————

【A 类｜完整性问题（Completeness）】
- 定义：在工程管理、合同管理、财务或档案制度中，明确要求必须存在的资料或单据，在被检查资料范围内【完全不存在任何实体或电子形式】的记录，包括原件、复印件、扫描件或系统记录。
- 唯一判定锚点：是否能看到、找到或调取到该资料本身？
- 判定标准：
    - 被检查范围内完全找不到该资料
    - 不存在纸质件、电子件、扫描件或系统记录
- 典型特征：缺、无、未提供、未提交、未见、未附、系统中不存在、档案中无此资料
- 排他规则：若资料存在但不规范、不一致或逻辑异常，则不属于 A 类。
- 明确排除：
    - 资料存在但未签字 → 不属于 A 类
    - 资料存在但未盖章 → 不属于 A 类
    - 资料存在但格式不规范 → 不属于 A 类

【B 类｜规范性问题（Formal Compliance）】
- 定义：资料或单据【可以被看到、找到或调取到】，但其填写要素、格式或形式不符合制度、合同或管理要求。
- 唯一判定锚点：资料是否“客观存在，但填得不合格”？
- 判定标准：
    - 资料实体或电子文件客观存在
    - 问题仅涉及填写要素、格式或形式缺陷
    - 不涉及真实性怀疑、不涉及业务逻辑错误
- 典型特征：未填写日期、未签字、未盖章、编号缺失、格式错误、使用非规定表单、以其他单据替代
- 明确排除：
    - 完全没有该资料 → 不属于 B 类
    - 单据存在明显涂改 → 不属于 B 类
    - 时间顺序明显不合理 → 不属于 B 类

【C 类｜一致性问题（Consistency）】
- 定义：同一工程事项，在两份或以上有效资料之间，存在数量、金额、内容、对象、累计结果不一致的情况。
- 判定标准：
    - 至少涉及两份资料
    - 每一份资料本身“形式上合规”
    - 问题核心为“相互对不上”
- 典型场景：送货单与入库单不一致；出库数量与劳务结算工程量不一致；多单位结算工程量合计超出实际。
- 排他规则：若不一致源于明显时间倒置或真实性存疑，则不属于 C 类。

【D 类｜逻辑性问题（Logical Validity）】
- 定义：资料虽存在且形式合规，但不符合正常工程实施顺序、业务流程或时间逻辑，即使数量一致，仍无法成立。
- 判定标准：
    - 重点在“时间先后”“流程顺序”
    - 不要求数量不一致
    - 不要求资料缺失
- 典型场景：未中标先采购；未开工先结算；一天内出库即完成安装；工期结束后仍大量出库。
- 排他规则：若问题核心是数量不一致或资料缺失，则不属于 D 类。

【E 类｜合规性 / 程序性问题（Procedural Compliance）】
- 定义：工程管理行为违反合同约定、制度条款或审批流程，但相关资料可能齐全、规范、真实。
- 判定标准：
    - 是否违反约定或程序
    - 不讨论资料真假
    - 不讨论资料是否存在
- 典型场景：超期签订合同；合同约定违约金但未执行；工程延期未履行审批；工程变更发生在开工令之前。
- 排他规则：若问题本质为资料缺失或逻辑异常，则不属于 E 类。

【F 类｜真实性 / 可靠性问题（Authenticity & Reliability）】
- 定义：资料存在，但存在明显迹象表明其真实性或可信度不足，包括涂改、篡改、日期异常、逻辑自相矛盾、技术性不可读取。
- 判定标准：
    - 真实性优先于一切其他判断
    - 一旦成立，其他类别不再考虑
- 典型场景：明显涂改；日期反复修改；签收早于送货；单据无法打开。
- 排他规则：F 类优先级最高。

【G 类｜非资料问题 / 不适用问题（Not Applicable）】
- 定义：问题描述不涉及工程资料的完整性、规范性、一致性、逻辑性、合规性或真实性，或信息不足以进行判断。
- 适用情形：管理建议、模糊描述、重复性说明、责任提醒、超出工程资料审查范围的问题。
- 使用限制：仅当 A–F 类均不适用时，方可选择 G 类，不得作为兜底类别。

——————————————————
【分类规则（必须遵守）】
——————————————————
1. 只能选择一个问题类别；
2. 禁止多选、禁止模糊判断；
3. 若同时符合多个特征，选择风险等级最高的类别（F > D > E > C > B > A > G）；
4. 不得自行新增、合并或拆分类别。

——————————————————
【输出要求（必须严格遵守）】
——————————————————

输出内容必须严格符合以下 JSON 结构（仅为结构示意，不得原样复制格式符号）：

{
  "category_code": "",
  "category_name": "",
  "judgement_reason": ""
}

规则说明：
1. 只允许输出一个合法的 JSON 对象；
2. 不得输出任何 JSON 之外的文字、说明、解释、注释或 Markdown 符号；
3. JSON 仅允许包含以下三个字段，不得新增、删除或改名字段：
   - category_code
   - category_name
   - judgement_reason
4. category_code 只能是 "A" "B" "C" "D" "E" "F" "G" 之一；
5. category_name 必须与 category_code 一一对应；
6. judgement_reason 用一句话说明核心判定依据，不超过 30 个汉字；
7. 若输出不符合以上任意一条，视为错误输出。
